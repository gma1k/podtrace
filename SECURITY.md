# Security Policy

## Supported Versions

Podtrace is pre-1.0. Only the most recent released minor line and
`main` receive security backports.

| Version  | Supported          |
|----------|--------------------|
| main     | ✅                 |
| 0.11.x   | ✅                 |
| < 0.11.0 | ❌                 |

When a new minor line ships, the previous one's support window ends.
Pre-1.0 cadence is intentionally tight to keep the maintained surface
small.

## Reporting an Issue

We strongly encourage you to report security issues **privately**
before disclosing them in any public forum (GitHub issues, social
media, mailing lists, conference talks, etc.).

Send reports to:

**ghassan+podtrace@malke.nl**

with a subject line starting with `[podtrace-security]`. Plain text
is fine; PGP encryption is welcome but not required.

Please include:

- A clear description of the issue and its impact.
- Steps to reproduce, ideally with a minimal example or PoC.
- The podtrace version (`podtrace --version`), Kubernetes version,
  kernel version, and host distribution.
- Any patch or mitigation you've already developed.

You can expect:

- **Acknowledgement within 3 business days.** If you don't hear back
  within that window, please re-send — emails do occasionally get
  lost.
- **Triage and confirmation within 7 business days.**
- **Coordinated disclosure timeline** agreed with you. The default
  is a 90-day private window, sooner if a fix ships earlier.
- **Reporter credit** in the release notes (omit if you prefer to
  remain anonymous).

## Scope

In scope:

- Issues in the podtrace binary, operator, agent, or BPF programs.
- Issues in the published container image, Helm chart, or quickstart
  manifests at `ghcr.io/gma1k/podtrace`, `ghcr.io/gma1k/charts/podtrace`,
  and `releases/*/quickstart.yaml`.
- Cosign signature spoofing, transparency-log gaps, or supply-chain
  issues affecting podtrace artifacts.
- Privilege escalation through podtrace's RBAC, validating webhooks,
  or session Job templates.

Out of scope:

- Issues in upstream Kubernetes, the host kernel's BPF verifier, or
  third-party exporters (OTLP, Jaeger, Splunk, DataDog, Zipkin) —
  please report those to their respective maintainers.
- Issues that require an attacker who already has cluster-admin
  privileges (these are not in our threat model).
- Denial-of-service via excessive trace events on the agent's ring
  buffer (rate-limiting is a user concern; podtrace exposes
  `samplePercent` and `thresholds` for this).
- Issues in development-only paths: `make build`, contributor
  scripts, the chainsaw test harness.

## Threat Model (in progress)

A formal threat model document is planned for podtrace's `v1beta1`
graduation. Until then, the principles below summarize what podtrace
assumes about its environment:

- The **operator** runs unprivileged (distroless nonroot, no host
  mounts). A compromised operator can only manipulate CRDs in its
  own namespace — it cannot escalate to host privileges directly.
- The **agent DaemonSet** runs with `CAP_BPF + CAP_PERFMON +
  hostPID` on each node. A compromised agent is roughly equivalent
  to a compromised privileged workload on that node.
- **Per-session Jobs** inherit the agent's capability set and read pod
  metadata via a per-session Role generated by the operator. A
  compromised session Job has the same blast radius as the agent on
  that one node.
- **Cross-namespace tracing is consent-gated.** A namespaced CR
  (`PodTrace`, `PodTraceSession`, `PodTraceSchedule`) may only target
  pods in a foreign namespace when that namespace opts in with the
  `podtrace.io/allow-tracing-from` annotation. The operator enforces
  this at every reconcile (the authoritative gate) and the validating
  webhook enforces it at admission (fast feedback). Without the grant,
  a CR is confined to its own namespace, it cannot capture another
  tenant's decrypted L7 traffic, DNS, syscalls, or file paths by naming
  their namespace. See
  [docs/cross-namespace-cr-targeting.md](docs/cross-namespace-cr-targeting.md#target-namespace-consent-required).
- **All published artifacts** (image, chart, tarballs) are
  cosign-signed keyless via GitHub OIDC, with the signing event
  recorded in the public Sigstore Rekor transparency log. Verify
  the chain before running in trusted environments — see
  [docs/installation.md](docs/installation.md) for the verification
  commands.

## Hardening Notes

For users running podtrace in environments where attacker capability
is a concern:

- Pin to a specific image **digest** (`ghcr.io/gma1k/podtrace@sha256:...`)
  rather than a tag, after verifying the cosign signature.
- Restrict the namespaces where users can create `PodTrace` and
  `PodTraceSession` CRs via Kubernetes RBAC. Delegating `create` on
  these CRs in a namespace is safe by default: consent-gating confines
  the delegated authority to that namespace unless a *target* namespace
  explicitly opts in via `podtrace.io/allow-tracing-from`.
- Grant `podtrace.io/allow-tracing-from` narrowly — prefer an explicit
  list of trusted observer namespaces over `*`, and treat the wildcard
  as equivalent to letting any CR author in the cluster intercept that
  namespace's in-flight traffic.
- Cap session capacity per node via `TracerConfig.spec.maxConcurrentSessionsPerNode`.
- Use the `ExporterConfig.spec.samplePercent` cap to bound the agent's
  data-plane load.

For per-distro hardening notes (SELinux, AppArmor, Pod Security
Standards), see the platform guides under [docs/](docs/).
